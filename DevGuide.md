# 开发指南

这份文档是 Lumina 当前阶段的接手说明。

目标不是写成一份简略介绍，而是让下一个开发者或 Agent 在尽量少猜测的前提下，快速理解：

- 项目现在已经做到什么程度
- 当前设计为什么这样写
- 哪些地方是临时方案，哪些地方是明确的长期方向
- 下一步应该优先做什么，不应该急着做什么

---

## 1. 项目定位

Lumina 是一个基于 `adbutils + OpenCV` 的 FGO 自动化脚本。

当前阶段的定位非常明确：

- 先优先适配 MuMu 模拟器
- 先保证一条完整刷本链路稳定跑通
- 先用简单直接、容易调试的方案完成闭环
- 暂时不追求多设备适配
- 暂时不追求“高智能”战斗策略

换句话说，当前项目不是通用自动化框架，也不是成熟产品，更像是一个围绕固定刷本流程逐步演化的自动化工具。

---

## 2. 当前已实现的完整链路

截至当前版本，项目已经具备以下基础刷本链路：

1. 在主菜单识别关卡入口
2. 按配置点击固定关卡位
3. 进入助战选择界面
4. 按配置选择职阶
5. 识别目标助战从者模板，支持滑动搜索
6. 搜索失败时支持点击列表更新并再次搜索
7. 仍失败时回退到默认助战位
8. 进入编队确认界面
9. 点击开始任务
10. 进入加载提示界面，等待提示消失并额外缓冲
11. 进入战斗可操作界面
12. 首回合按 YAML 顺序释放从者技能 / 御主技能
13. 技能或宝具若触发目标选择，默认选择第一个从者
14. 后续回合只执行攻击，不重复释放技能
15. 进入选卡界面后优先判断宝具是否可释放
16. 可释放的宝具优先出卡，再用普通指令卡补足 3 张
17. 进入结算界面后按模板完成两次“点击屏幕”与“下一步”
18. 回到主菜单后，若 `loop_count` 未达上限，则继续下一轮

这是当前项目最重要的主链路。后续扩展都应该以“不破坏这条链路”为前提。

## 4. 当前识别机制说明

这是接手时最容易误解的地方。

### 4.1 每轮主循环只截图一次

当前机制不是“每个状态单独截图一次”，而是：

1. 主循环开始
2. 截图一次
3. 这张截图给所有状态模板共用
4. 识别出当前状态
5. 执行 handler
6. 下一轮再重新截图

这意味着很多“识别异常”并不是模板本身坏了，而是“那一轮截图时机不对”。

### 4.2 未知状态截图

当流程进入 `UNKNOWN` 时：

- 当前截图会自动保存到 [assets/screenshots/unknown](/D:/VSCodeRepository/Lumina/assets/screenshots/unknown)
- 日志里会输出最佳候选状态和匹配分数

这是当前排查问题最重要的手段之一。

### 4.3 模板来源建议

当前模板匹配对像素一致性要求非常高。

结论非常明确：

- 最稳的模板来源：直接从 MuMu 实际游戏截图中手工裁切
- 不推荐直接使用 atlas / 素材站下载图做模板

原因不是素材站一定错，而是当前 `matchTemplate` 算法不看“语义相同”，只看“像素是否足够一致”。

即使你肉眼觉得 atlas 下载图和游戏截图一模一样，也可能因为：

- 缩放方式不同
- 抗锯齿不同
- 外框、阴影、透明边不同
- 裁切边界不同

导致匹配分数显著下降。

所以当前项目的模板工作规范应该是：

- UI 模板优先从实际运行截图裁
- 助战头像模板也优先从实际运行截图裁
- 不要急着把 atlas 素材强行塞进当前匹配逻辑

## 5. 当前配置结构

主配置文件：

- [config/battle_config.yaml](/D:/VSCodeRepository/Lumina/config/battle_config.yaml)

当前主要字段：

### 5.1 刷本主流程

- `loop_count`
  完整战斗完成次数，`-1` 表示无限循环

- `quest_slot`
  主菜单固定关卡入口编号，当前支持 `1 / 2 / 3`

### 5.2 助战选择

- `support.class`
  当前支持：
  - `all`
  - `berserker`

- `support.servant`
  目标助战从者英文目录名，例如：
  - `morgan`
  - `castoria`

- `support.pick_index`
  当未找到目标从者时，默认回退点击的助战位，当前支持 `1 / 2 / 3`

- `support.max_scroll_pages`
  最多滑动搜索的页数

### 5.3 战斗动作

- `skill_sequence`
  首回合要释放的动作序列

当前兼容两种写法：

- 纯数字 `1-9`
  表示从者技能

- 结构化动作
  例如：

```yaml
- type: master
  skill: 1
```

表示御主技能

### 5.4 调试

- `log_level`
  调试时建议使用 `DEBUG`

- `match_threshold`
  模板匹配阈值

- `skill_interval`
  技能释放后的额外等待时间

- `save_debug_screenshots`
  是否在刷新时落盘普通截图

## 6. 当前已实现的重要设计决策

这部分很关键，因为很多后续工作必须建立在这些决策之上。

### 6.1 主菜单不是结束状态，而是刷本入口状态

之前的逻辑里，识别到主菜单就直接结束。

现在已经改成：

- 主菜单 = 可以点击关卡入口的起始状态
- 每轮刷本结束后，如果次数没到，就回到主菜单继续发起下一轮

### 6.2 助战选择优先做“可运行”，不是“最完整”

当前助战逻辑的优先级是：

1. 职阶筛选
2. 当前页模板搜索
3. 向上滑动继续搜索
4. 超页数后点击列表更新
5. 再搜索一次
6. 仍失败则回退默认位

这个设计目标是“不因目标助战暂时没找到就让整场刷本直接报废”。

### 6.3 技能只在首个可操作回合释放

当前战斗逻辑明确区分：

- 首次进入 `BATTLE_READY`
  执行完整技能动作序列，然后攻击

- 后续再次进入 `BATTLE_READY`
  不再重复放技能，只点攻击

这是为避免后续回合重复放技能导致逻辑错位。

### 6.4 宝具逻辑当前只做最小可用版本

当前版本不做：

- OCR 读 NP 数值
- 宝具优先级策略优化
- 宝具与颜色链混合策略
- 单体/群体宝具分类

当前只做：

- 用固定 NP 区域亮度近似判断“是否可放宝具”
- 优先点击可用宝具卡
- 不足 3 张时再补普通卡
- 若宝具触发目标选择，则默认选第一个从者

这里要明确：

`NP_REGIONS` 和阈值目前只是工程初版，不是已校准数据，需要基于真实截图重新标定。

### 6.5 资源目录开始分层，但还未完成完全迁移

当前资源结构已经开始过渡：

- `assets/ui/...`
- `assets/states/...`
- `assets/servants/<servant_name>/support/portrait.png`

但还不是完全迁移状态。

当前 `ResourceCatalog` 的目标是：

- 支持新结构
- 同时兼容旧的平铺路径

所以后续不要急着一次性重构所有模板，优先遵循兼容迁移原则。

## 7. 当前最容易出问题的地方

### 7.1 助战模板命中失败

常见原因：

- 模板不是从实际游戏截图裁的
- 当前页头像带有选中高亮/边框变化
- 模板只裁了纯头像，但实际界面还有职业角标或礼装遮挡
- 滑动停留位置不稳定，导致头像实际像素位置和截图状态发生变化

优先排查：

- 模板是否来自 MuMu 实际截图
- 目标页未知状态截图
- `DEBUG` 日志里的模板匹配分数

### 7.2 战斗回合里不动作

优先看：

- 是否识别到了 `BATTLE_READY`
- 是否进入了 `CARD_SELECT`
- `handle_battle_ready()` 是否重复释放技能或只攻击
- 选卡后的等待是否足够

### 7.3 宝具误判

当前宝具判定是非 OCR 方案，风险点天然存在：

- NP 条区域选错
- 阈值不合适
- 某些战斗特效或 UI 高亮干扰亮度均值

所以这块后续大概率还要继续校准。

## 9. 当前状态总结

如果只看“项目是否能用”，当前答案是：

- 可以用于 MuMu 上的固定刷本流程
- 能跑一条基础战斗链路
- 能根据配置释放从者技能与御主技能
- 能做助战模板搜索与回退
- 能做宝具优先的基础出卡

如果只看“项目是否成熟”，当前答案是：

- 还远未成熟
- 仍然高度依赖固定模板、固定坐标、固定场景
- 战斗内智能度还比较低
- 大量关键参数仍需要手工校准

所以对当前状态的正确判断是：

这是一个已经拥有清晰骨架和可用主链路的自动化原型，而不是已经完成的系统。

## 10. 未来版本规划

这里按优先级而不是按功能模块罗列。

### v4 优先方向：选卡智能化

建议优先做：

- 红 / 绿 / 蓝指令卡识别
- 普通卡类型判定
- 同色链基础优先级
- 助战从者优先级

这会直接提升战斗内收益，而且不需要立刻引入 OCR。

### v5 优先方向：技能智能化

建议后续做：
