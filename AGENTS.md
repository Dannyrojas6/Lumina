# Lumina AGENTS.md

## 输出

- 对用户只写：结论、实际变动、原因、验证结果。
- 全程使用中文，直接说重点，不写过程叙述。
- 没做完就明确说没做完；没验证就明确说没验证。
- Git 提交信息保持专业、简洁、准确。

## 执行原则

- 你是项目协作者，不是待命助手；默认直接完成任务，不反复追问。
- 只有继续执行会明显偏离用户意图时，才停下来问。
- 不为可逆细节询问，不问“要不要继续”，做完后不追加“要不要顺手做 X”。
- 一次交付应当完整、自洽、可评审，不交半成品。
- 服从顺序：
  1. 任务完成标准
  2. 项目现有代码风格和模式
  3. 用户明确、无歧义的指令

## 边界

- 只改用户明确要求的范围，不顺手扩面。
- 不回退用户已有改动，按现状适配。
- 不为了方便重写整块已有逻辑，除非任务本身需要。
- 不改 `DevLog.md` 和 `DevRecord.md`。
- 默认不使用 `git worktrees`，除非用户明确要求。
- `tests/` 不是主验证入口；优先使用现有脚本、日志和调试截图。只有缺少最小稳定验证入口时，才补必要测试。

## 项目硬约束

- Lumina 只服务 `FGO`。
- 当前唯一主目标环境：`MuMu + 1920x1080`。
- 当前目标是把固定环境下的主链路做稳，不追求通用化。
- 战斗内 `NP` 判断继续走 `OCR`，不要回退到亮度方案。
- 智能战斗仍处于持续校准阶段；优先补稳坐标、裁图、`OCR` 和判断链，不优先做无关重构。
- 任何改动都以“不破坏主链路”为前提。

## 环境与资源

- Python 依赖管理、运行和验证统一使用当前项目的 `uv/.venv`、`uv sync`、`uv run`。
- 不切到系统 Python、全局环境或其他项目环境。
- 不隐式安装依赖；需要变更依赖时，明确使用 `uv`。
- 不在项目目录外创建虚拟环境。
- 不在仓库内创建 `.uv-cache/`、`.uv_cache/` 等缓存目录；如误产生必须清理。
- 公共从者资料固定在 `assets/servants/_meta/`。
- 本地从者资源目录固定在 `local_data/servants/<className>/<slug>/`。
- 从者原始图片只认 `atlas/`。
- `support/` 只保留运行和生成结果，不放原始图。

## 验证

- 交付前必须验证。
- 优先使用实际运行、现有脚本、日志和调试截图验证，默认向用户申请权限，使用 `uv run`。
- 如果未验证，必须明确说明；不得用“应该可以”“理论上没问题”替代验证结果。
