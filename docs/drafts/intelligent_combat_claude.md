# FGO 智能战斗系统设计文档

## 文档概述

本文档旨在将"完全模拟人类对 FGO 战局做出正确判断"这一目标，拆解为可落地的系统设计结构。核心理念是：**不追求数学最优解，而是追求"像人一样思考"的决策过程**。

------

# 第一章：设计哲学

## 1.1 人类判断 vs 机器判断的本质差异

在开始设计之前，必须先明确我们要模拟的对象究竟是什么。

人类玩家在 FGO 中的决策**从来不是精确计算**，而是一种**基于经验积累的模糊直觉**。具体表现为：

- 人类不会计算精确伤害值，而是判断"大概能不能打死"
- 人类不会枚举所有手牌组合，而是"一眼看出"哪张卡最重要
- 人类会根据情绪和风险偏好做出"赌一把"或"稳一手"的选择
- 人类对敌方行为有"预感"，本质是对敌方 AI 脚本规律的经验积累

因此，本系统的设计目标不是"最优解引擎"，而是**"合理解决策引擎"**——在绝大多数情况下做出不输给中高水平人类玩家的判断。

## 1.2 核心设计原则

**原则一：分层决策，而非扁平穷举** 人类不会同时考虑所有因素，而是有优先级地逐层过滤。系统也应如此。

**原则二：模糊评估优于精确计算** 系统应使用"伤害区间"而非"精确伤害值"，使用"高/中/低"风险而非"XX%概率"。

**原则三：宏观战略驱动微观操作** 先决定"这回合我要干什么"（宏观意图），再决定"用什么牌和技能实现它"（微观执行）。

**原则四：状态感知先于行动选择** 必须先充分理解当前局势，才能进行任何决策。感知是决策的前提，不可跳过。

------

# 第二章：系统总体架构

## 2.1 架构总览

整个智能战斗系统由五个核心模块组成，形成一个完整的决策闭环：

```
┌─────────────────────────────────────────────────────┐
│                   战场信息输入层                      │
│        （血量、NP、技能CD、Buff状态、手牌）            │
└────────────────────┬────────────────────────────────┘
                     │
                     ▼
┌─────────────────────────────────────────────────────┐
│              模块一：局势感知引擎                     │
│       将原始数据转化为"有意义的局势描述"              │
└────────────────────┬────────────────────────────────┘
                     │
                     ▼
┌─────────────────────────────────────────────────────┐
│              模块二：战术意图识别器                   │
│        判断当前局势属于什么"战斗阶段"                 │
└────────────────────┬────────────────────────────────┘
                     │
                     ▼
┌─────────────────────────────────────────────────────┐
│              模块三：行动候选生成器                   │
│      根据意图，生成有限个值得考虑的行动方案           │
└────────────────────┬────────────────────────────────┘
                     │
                     ▼
┌─────────────────────────────────────────────────────┐
│              模块四：多维评分引擎                     │
│         对每个候选方案进行综合打分排序                │
└────────────────────┬────────────────────────────────┘
                     │
                     ▼
┌─────────────────────────────────────────────────────┐
│              模块五：执行决策输出层                   │
│         输出最终的技能释放顺序 + 出牌顺序             │
└─────────────────────────────────────────────────────┘
```

这五个模块的关系是**串行过滤**，而不是并行计算。每个模块负责缩小问题范围，将复杂度逐步降低。

# 第三章：模块一——局势感知引擎

## 3.1 模块职责

这是整个系统最基础也最关键的模块。它的职责是将战场上的**原始数值数据**，转化为系统可以"理解"的**语义化局势描述**。

人类看到"Boss 还剩 30% 血量"时，脑海中浮现的不是数字本身，而是"快打穿第一条 Break 了"的判断。这个转化过程就是局势感知引擎要做的事。

## 3.2 敌方状态感知

### 3.2.1 生命值语义化

对每个敌方单位的血量进行语义标注，而不是保留原始数值：

- **安全区**：距离下一个 Break 条还有较大余量，当前输出压力不紧迫
- **临界区**：即将触发 Break 条，需要在本回合或下回合打穿
- **Break 后**：已触发 Break，需评估新出现的 Break 效果是否带来新威胁
- **濒死区**：本回合集火即可击杀

每个区间的划分不依赖固定百分比，而是依赖**距离 Break 点的绝对伤害差值**与**我方当前输出能力**的比值。

### 3.2.2 NP 充能威胁评估

敌方 NP 充能是最核心的时间压力来源。系统需要对每个敌人的 NP 状态进行威胁分级：

- **无威胁**：NP 尚未开始充能，或充能极低
- **潜在威胁**：充能达到中段，若不加干预，将在 2-3 回合内发动
- **即时威胁**：充能接近满，下一回合大概率发动宝具
- **宝具锁定**：充能已满，本回合必然发动（需要无敌/回避应对）

### 3.2.3 敌方行动脚本识别

FGO 的 Boss 并非随机行动，而是按照固定脚本执行。系统应维护一个**已知关卡的 Boss 行动数据库**，记录：

- 每回合固定使用的技能种类（强化/弱化/特殊效果）
- Break 条触发时的固定行为（清空我方 Buff、施加全体 Debuff 等）
- NP 发动的固定条件（每 X 回合、血量低于 Y% 时等）
- 特殊机制的触发规律（如属性吸收、行动不能等）

对于未知关卡，系统应在前几回合进入**观察积累模式**，记录敌方行为规律，逐步建立预测模型。

## 3.3 己方状态感知

### 3.3.1 从者战斗力评估

对每位从者进行当前实际战斗价值的综合评估：

**输出评估维度：**

- 当前手牌中持有的色牌数量与质量
- 宝具充能进度（区分"本回合可发"/"下回合可发"/"需要积累"）
- 当前生效的攻击强化 Buff 的剩余回合数
- 克制关系带来的伤害加成

**生存评估维度：**

- 当前血量与最大血量的比值
- 是否持有无敌/回避等防御性 Buff
- 是否处于无法行动状态（魅惑、恐慌、行动不能）
- 礼装的防御性效果是否已触发/剩余次数

### 3.3.2 技能资源评估

技能是人类玩家最重要的决策对象之一，系统需要对每个技能的"现在用 vs 留着用"进行评估：

对每个技能标注以下属性：

- **即时价值**：现在使用能带来多少收益
- **时机价值**：是否存在一个"更好的使用时机"（如等 Buff 叠满再用）
- **紧迫度**：是否必须在本回合使用，否则会造成损失
- **冷却代价**：使用后进入冷却，在关键时刻无法使用的风险

### 3.3.3 手牌结构分析

手牌是每回合决策的直接操作对象：

- 统计各色牌（Buster/Arts/Quick）的数量分布
- 识别可以形成链的组合（三张同色、首张同色等）
- 识别哪位从者在当前手牌中最具价值（宝具即将满 + 持有 Arts 牌）
- 评估当前手牌是否"值得"出完，还是应该"钻空子"保留资源

## 3.4 局势综合描述输出

局势感知引擎最终输出一个结构化的**局势快照**，供后续模块使用。该快照包含：

- 当前最大威胁来源（是哪个敌人，威胁类型是什么）
- 我方最强输出点（哪位从者本回合爆发价值最高）
- 资源紧张度（NP/技能/星星 各维度是否充裕）
- 时间压力等级（距离局势恶化还有几回合的容忍空间）

# 第四章：模块二——战术意图识别器

## 4.1 模块职责

这是最接近"人类思维模式"的模块。人类玩家在开始想具体操作之前，会先形成一个**本回合的主要目标**，例如"这回合要把 Boss 打穿第一条血条"或"先把无敌挂上撑过宝具"。

战术意图识别器就是负责形成这个**宏观目标**的模块。

## 4.2 战斗阶段定义

系统将战斗的整体流程划分为若干**宏观阶段**，每个阶段对应不同的优先级逻辑：

### 阶段一：前期积累阶段

**触发条件**：己方资源不足（NP 低、关键技能在 CD 中）、且敌方威胁不紧迫

**核心目标**：

- 高效充能 NP，准备爆发
- 积累暴击星，等待最优输出时机
- 挂上持久性 Buff，为后续爆发做铺垫
- 消耗敌方危险的充能速度（通过攻击减速或 NP 充能削减技能）

**决策倾向**：优先打 Arts 链积累 NP，不急于消耗高价值资源，技能优先用持久型而非爆发型

### 阶段二：爆发准备阶段

**触发条件**：关键从者宝具接近满充、或敌方 Break 条进入临界区、或时间窗口即将关闭

**核心目标**：

- 叠加爆发前的最后一层强化 Buff
- 确保爆发回合的手牌最优（宝具 + Buster 链）
- 使用提升 NP 充能速度的技能，确保本回合或下回合宝具齐发

**决策倾向**：开始消耗技能，为爆发回合布局，可以接受本回合输出稍低

### 阶段三：爆发执行阶段

**触发条件**：宝具已满 + 强化 Buff 已到位 + 敌方处于临界/濒死区

**核心目标**：

- 打出最高单回合伤害
- 优先击杀高威胁目标
- 尽量打穿 Break 条，减少敌方获得行动机会

**决策倾向**：全力输出，无条件使用所有爆发性资源，不为"下一回合"保留任何东西

### 阶段四：防御应对阶段

**触发条件**：敌方宝具即将发动、或己方血量危险、或关键从者即将死亡

**核心目标**：

- 挂上无敌/回避，防御敌方宝具
- 优先治疗濒死从者
- 必要时考虑复活顺序和换人时机

**决策倾向**：输出完全让步于生存，宁可浪费输出手牌也要确保关键从者存活

### 阶段五：收尾清场阶段

**触发条件**：敌方血量极低，本回合普攻即可解决

**核心目标**：

- 尽快结束战斗，避免不必要的风险
- 提前为下一波或下一关储备资源

**决策倾向**：不浪费宝具打已经必死的敌人，将资源留到下一波

## 4.3 阶段切换逻辑

战斗阶段之间的切换不是线性的，而是**随时响应**的：

- 正处于积累阶段时，若检测到敌方宝具即时威胁 → 立即切换至防御应对阶段
- 正处于爆发准备阶段时，若关键从者突然濒死 → 立即切换至防御应对阶段
- 防御阶段处理完毕（敌方宝具已撑过）→ 回归之前的阶段继续执行

切换优先级从高到低：**防御应对 > 爆发执行 > 爆发准备 > 收尾清场 > 前期积累**

## 4.4 复合意图处理

人类在战斗中经常同时有多个目标，例如"这回合既要打穿 Break 条，也要挂上无敌防下回合宝具"。

系统需要支持**意图的优先级叠加**，而不是只允许单一意图：

- 主意图：本回合最核心的目标
- 次意图：在不影响主意图的前提下额外追求的目标
- 约束条件：无论如何都不能违反的底线（如"关键从者不能死"）

# 第五章：模块三——行动候选生成器

## 5.1 模块职责

根据战术意图，生成**有限个、有意义的**行动候选方案。

这里的关键设计思想是：**不枚举所有可能性，而是只生成"人类会考虑的方案"**。

人类面对手牌时，不会逐一思考所有排列组合。他们会直接过滤掉明显次优的选项，只在少数几个"看起来合理"的方案之间权衡。系统也应该这样工作。

## 5.2 技能使用候选生成

### 5.2.1 技能的分类框架

将所有技能按照功能分为以下类别，每类技能的使用时机逻辑不同：

**即时爆发类**（如攻击力强化、宝具强化）：

- 使用时机：即将发动宝具的前一个步骤
- 禁止时机：宝具还需要 2 回合以上才能发动时

**资源生成类**（如 NP 充能、星星生成）：

- 使用时机：资源紧张、需要快速充能时
- 考量点：用完后距离下一次可用的 CD 是否会影响关键时刻

**持久性强化类**（如多回合 Buff）：

- 使用时机：越早用越好，让 Buff 效果最大化
- 考量点：是否有更重要的时机需要等待

**防御保命类**（如无敌、回避、治疗）：

- 使用时机：敌方宝具将至、或队员濒死
- 禁止时机：局势安全时不应浪费

**控制干扰类**（如沉默、魅惑、即死）：

- 使用时机：敌方宝具充能高、或需要延缓敌方行动时
- 考量点：目标是否对该控制效果免疫

### 5.2.2 技能组合候选生成规则

系统不逐一考虑每个技能，而是按以下规则生成有意义的技能组合：

**规则一（强制执行）**：检测是否有"必须本回合使用"的技能，将其纳入所有候选方案

**规则二（意图驱动）**：根据当前战术意图，筛选与意图匹配的技能类别

**规则三（协同识别）**：识别技能之间的协同关系（如先用充能技能，再用攻击强化，再发宝具），将协同组合作为一个整体候选

**规则四（冲突排除）**：排除逻辑冲突的技能组合（如对同一目标叠加两个相同类型的强化，但该类型有上限时）

## 5.3 出牌顺序候选生成

### 5.3.1 出牌候选筛选原则

从所有手牌的可能出牌组合中，按以下逻辑筛选出值得考虑的候选：

**首要过滤——宝具时机过滤**：

- 如果本回合有从者宝具可发且意图为爆发，宝具必须出现在候选方案中
- 宝具在出牌序列中的位置会显著影响输出效果（宝具前的色卡决定链类型）

**次要过滤——色链价值过滤**：

- 优先生成能形成同色链的候选（三张同色 > 首张同色 > 混色）
- 在意图为积累时，优先生成含多张 Arts 牌的候选
- 在意图为爆发时，优先生成含多张 Buster 牌的候选

**辅助过滤——星星分配过滤**：

- 当手中暴击星充足时，优先考虑将星星分配给最高价值从者的候选
- 出牌顺序会影响星星的分配方向，需考虑"哪张牌放在哪个位置"

### 5.3.2 候选方案数量控制

系统最终生成的候选方案数量应控制在 **3 到 7 个之间**。

少于 3 个：可能遗漏重要方案 多于 7 个：评分模块的计算量过大，且超出了人类的真实决策范围

# 第六章：模块四——多维评分引擎

## 6.1 模块职责

对候选方案进行综合打分，选出最优方案。

这是系统中最需要精细调参的模块，因为各评分维度的**权重比例**直接决定了系统的"风格"——是激进的还是保守的，是输出导向的还是生存导向的。

## 6.2 评分维度体系

评分系统分为**核心维度**和**修正维度**两层。

### 6.2.1 核心评分维度

**维度 A：威胁消除得分**

衡量该方案对当前最大威胁的处理效果：

- 能否在本回合打穿即将 Break 的血条
- 能否击杀即时威胁等级的敌人
- 能否阻止或延缓敌方宝具的发动

这是**权重最高**的维度，因为"防止局势恶化"永远优先于"追求最大收益"。

**维度 B：输出效率得分**

衡量该方案的伤害输出质量：

- 色链的类型与完整性（完整同色链 > 部分链 > 混色）
- 宝具的使用与强化 Buff 的配合程度
- 暴击星的有效利用率（分配给了高价值从者）
- 克制属性带来的额外加成是否被充分利用

**维度 C：资源代价得分**

衡量该方案消耗的资源是否合理：

- 使用的技能的时机合理性（在正确的时机用对的技能）
- NP 消耗是否划算（发动宝具的时机是否是最高价值时机）
- 使用高 CD 技能后，是否会在关键时刻面临"无技能可用"的窘境

**维度 D：未来预期得分**

衡量该方案对下一回合乃至未来局势的铺垫：

- 本回合后 NP 充能的预期积累量
- 本回合后暴击星的预期余量
- 关键技能的 CD 恢复进度
- 执行该方案后，下回合的手牌期望质量

**维度 E：风险规避得分**

衡量该方案的安全性：

- 是否确保了关键从者的存活
- 是否为敌方宝具预留了应对手段
- 是否存在"赌博性"操作（如依赖随机触发效果）
- 若本方案失败，是否有兜底方案

### 6.2.2 修正评分维度

这些维度对核心得分进行加成或惩罚：

**修正 F：意图契合度** 方案与当前战术意图的吻合程度。高度吻合的方案获得加分，背离意图的方案获得惩罚。

**修正 G：资源浪费惩罚** 如果方案存在明显的资源浪费（如强化 Buff 叠加上限溢出、击杀已经濒死的敌人时使用了宝具等），扣除对应分数。

**修正 H：协同加成** 方案中的技能与出牌产生明显协同效果时（如充能技能 + 立即发动宝具），额外加分。

## 6.3 权重动态调整机制

这是本模块最重要的设计：**各维度的权重不是固定的，而是随当前战斗阶段动态调整的**。

| 战斗阶段 | 威胁消除 | 输出效率 | 资源代价 | 未来预期 | 风险规避 |
| -------- | -------- | -------- | -------- | -------- | -------- |
| 前期积累 | 中       | 低       | 高       | 高       | 中       |
| 爆发准备 | 中       | 中       | 中       | 高       | 低       |
| 爆发执行 | 高       | 高       | 低       | 低       | 低       |
| 防御应对 | 低       | 低       | 低       | 低       | 极高     |
| 收尾清场 | 高       | 中       | 高       | 高       | 低       |

这种动态权重机制，正是"人类根据局势调整打法"这一行为的系统化表达。

## 6.4 评分的模糊化处理

所有评分不使用精确数值，而使用**区间分级**：

- 极高收益 / 极低收益
- 高收益 / 低收益
- 中等收益 / 无明显收益
- 负收益

这种模糊化处理有两个好处： 一是更接近人类"大概能打死"的思维模式； 二是避免因游戏内数值的细微波动（如随机伤害浮动）导致错误的精确计算。

# 第七章：模块五——执行决策输出层

## 7.1 模块职责

将评分最高的候选方案转化为**具体的、有时序的操作指令序列**。

## 7.2 操作时序的重要性

FGO 中操作的执行顺序至关重要，同样的技能和卡牌，顺序不同效果可能截然不同：

- 先用充能技能，再发宝具 vs 先发宝具再充能（前者正确）
- 先用攻击强化，再出 Buster 牌 vs 先出牌再强化（前者正确）
- 先打普通攻击形成色链，再追加宝具 vs 先发宝具再普通攻击（顺序影响链类型）

因此，输出层需要解决的核心问题是：**将"用什么"转化为"按什么顺序用"**。

## 7.3 操作序列生成规则

**第一步：处理强制前置操作** 某些技能必须在特定操作之前完成（如无敌必须在敌方宝具到来前使用）。这类操作无条件排在序列最前端。

**第二步：确定宝具位置与前置色链** 如果本回合要发宝具：

- 根据期望触发的链类型，在宝具前安排对应色牌
- 宝具前通常放 1-2 张对应色牌以触发链效果
- 宝具后的牌用于补充资源（Arts 补 NP、Quick 补星）

**第三步：排列强化技能的释放时机** 攻击类强化技能应在其强化对象即将行动之前释放，不宜过早（避免被敌方 Debuff 清除）也不宜过晚（确保作用于本回合的行动）。

**第四步：处理目标选择** 在多个敌人存在的情况下，为每张卡和每个宝具指定攻击目标：

- 宝具优先打威胁最高的目标
- 普攻牌优先打最接近 Break 点的目标（集中火力）
- 在目标已确定必死的情况下，将余量伤害转移到下一威胁目标

**第五步：输出最终操作序列** 以有序列表的形式输出完整操作，包含：

- 每个技能的施放时机和作用目标
- 每张手牌的出牌顺序和攻击目标
- 宝具的发动时机（在手牌序列中的位置）

# 第八章：特殊场景处理

## 8.1 多波次关卡的资源管理

FGO 中常见多波次战斗，每波结束后进入下一波。这带来了**跨波次的资源规划问题**：

系统需要在结束当前波次时，评估下一波次的预期难度，并据此决定：

- 本波次是否要为下一波次"留手"（不消耗所有技能）
- 宝具是否要在本波次"省"着用，留到下一波高威胁时使用
- 友情点召唤或礼装的使用时机

这要求系统具备**跨波次的全局视野**，而不仅仅关注当前波次的最优解。

## 8.2 未知关卡的探索模式

面对没有行动脚本记录的新关卡，系统需要进入一个特殊的**探索模式**：

- 第一回合：采取最保守的策略，观察敌方行为
- 前三回合：持续记录敌方每回合的行动、充能速度、技能使用规律
- 形成初步行动模式假设后：逐渐切换至正常决策模式
- 关键时刻仍保持保守：如果对敌方下一步行为不确定，优先选择风险规避型方案

## 8.3 极端局势的应急处理

当局势极度不利时（如多名从者死亡、敌方连续数回合宝具），系统需要启动**应急模式**：

- 放弃所有输出优化，全力追求生存
- 重新评估复活资源（魔法石等）的使用价值
- 评估"撑到关键从者宝具可用"是否现实
- 如果局势已无解，尽早判断并终止（避免无意义的操作消耗）

## 8.4 礼装与特殊机制的处理

某些礼装和从者具有特殊触发机制（如"首回合满 NP"、"特定条件爆发"等），这类机制需要独立的处理逻辑：

- 建立**特殊机制触发条件数据库**
- 在行动候选生成阶段，将特殊机制的触发条件纳入考量
- 为"创造特殊机制触发条件"这一行为单独评分

# 第九章：系统的边界与局限

## 9.1 系统无法模拟的人类行为

诚实地承认系统的局限性，是设计文档不可缺少的部分：

**情绪性决策**：人类有时会出于"试试看"的心理尝试非最优操作，这不在本系统模拟范围内。

**元游戏知识**：人类玩家可能因为了解游戏设计者的意图，而做出超出数值分析的判断（如"这关设计这么怪，肯定有什么隐藏机制"）。

**未知机制的应对**：第一次遭遇全新机制时，即使是人类玩家也会迷惑，系统同样无法完美处理。

**极限数值博弈**：高端玩家对精确伤害区间的掌握（如"正好把 Boss 打到 1 血触发 Break"），需要系统支持更精确的伤害计算，超出模糊评估的范畴。

## 9.2 系统需要持续维护的部分

- **Boss 行动脚本数据库**：随游戏更新持续补充
- **新从者技能数据**：每次新从者实装后更新
- **权重参数调优**：需要在实际测试中反复调整，使决策风格趋近于"高水平人类玩家"
- **新机制适配**：游戏引入新机制时，需要对应扩展评分维度

------

# 第十章：设计文档总结

## 10.1 核心设计思想的一句话总结

> **"先理解局势，再形成意图，再筛选方案，再综合评分，最后有序执行。"**

这五个步骤，正是一个经验丰富的人类玩家在每回合开始时，脑海中无意识走过的完整决策链路。本系统的全部设计，都是对这条链路的显式化、结构化表达。

## 10.2 各模块职责总览

| 模块           | 输入         | 输出                        | 核心价值                       |
| -------------- | ------------ | --------------------------- | ------------------------------ |
| 局势感知引擎   | 原始战场数据 | 语义化局势快照              | 将数值转化为"有意义的理解"     |
| 战术意图识别器 | 局势快照     | 当前战斗阶段 + 本回合主意图 | 形成"这回合干什么"的宏观目标   |
| 行动候选生成器 | 战术意图     | 有限个合理行动方案          | 将无限可能性收窄至人类决策范围 |
| 多维评分引擎   | 候选方案列表 | 排序后的方案列表            | 在合理方案中选出最优           |
| 执行决策输出层 | 最优方案     | 有序操作指令序列            | 确保正确的操作以正确的顺序执行 |

## 10.3 后续迭代方向

本文档描述的是系统的**第一版骨架设计**，后续可在以下方向进行深化：

- 引入**向前模拟**能力，对未来 2-3 回合的局势进行预演
- 建立**玩家风格参数**，允许系统模拟不同类型的人类玩家（激进型/保守型/效率型）
- 构建**关卡难度分级**，根据难度自动调整决策的谨慎程度
- 开发**自动化测试框架**，通过大量战斗测试对权重参数进行数据驱动的优化
