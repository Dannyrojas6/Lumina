# 当前项目功能实现审查

更新时间：2026-04-05

这份文档只基于当前仓库代码、配置和脚本入口做审查，不写未来方案，不替代 `README.md` 或 `DevGuide.md`。
目标是把“已经实现了什么”“哪些只是能跑通骨架”“哪些还明显依赖前置条件或存在缺口”拆开，方便人工逐项核查。

## 1. 总体结论

当前项目已经有一条从进本到结算回下一轮的完整主链，但它不是“全流程都稳定、全场景都覆盖”的状态，而是：

- 已经有可运行骨架
- 强依赖固定环境：`MuMu + 1920x1080`
- 强依赖模板匹配、固定坐标和固定裁图
- 战斗内判断明显偏保守
- 助战识别已经接入主流程，当前主链已切到遮挡排除后的双路向量核验
- 本地从者资源是很多能力能否运行的前提

按当前代码真实状态区分：

| 功能块 | 代码状态 | 当前真实边界 |
| --- | --- | --- |
| 主菜单进本到结算循环 | 已实现 | 主链闭环已存在 |
| 页面状态识别 | 已实现 | 主要靠模板匹配，误识别风险仍在 |
| 战斗内 `OCR` | 已实现 | 覆盖波次、敌人数、回合数、敌方 `HP`、前排三位 `NP` |
| 技能可用性判断 | 已实现 | 只覆盖前排九个技能位，偏保守 |
| `smart_battle` | 已实现 | 只规划前排从者技能，不规划御主技能 |
| 固定技能序列 | 已实现 | 只在第一次进入 `BATTLE_READY` 时执行一次 |
| 助战人物头像识别 | 已实现 | 主流程只保留遮挡排除后的双路向量核验 |
| 资源下载脚本 | 已实现 | 可补本地从者资源 |
| 助战长时观察脚本 | 已实现 | 可在助战页持续下滑并命中留证 |

## 2. 主流程状态机审查

### 2.1 当前实际处理的页面状态

当前主流程实际进入处理表的状态有 8 个：

- `MAIN_MENU`
- `SUPPORT_SELECT`
- `TEAM_CONFIRM`
- `LOADING_TIPS`
- `DIALOG`
- `BATTLE_READY`
- `CARD_SELECT`
- `BATTLE_RESULT`

`core/game_state.py` 里还有 `IN_BATTLE`，但当前主流程并没有处理这个状态，属于遗留位。

### 2.2 主流程每个阶段实际做了什么

| 阶段     | 当前行为                                                          | 依赖条件                                     | 风险点                       |
| ------ | ------------------------------------------------------------- | ---------------------------------------- | ------------------------- |
| 启动入口   | `main.py` 只负责调用 `core.app.run()`                              | Python 环境、配置文件                           | 入口极薄，问题都集中在装配层            |
| 运行装配   | 读取配置、建日志、建资源对象、连 `ADB`、初始化模板识别、初始化 `OCR`、按需初始化 `smart_battle` | `config/battle_config.yaml`、`adb`、本地从者资源 | 如果配置里填了从者，但本地资源缺失，会直接报错退出 |
| 状态识别   | 每轮先截图，再遍历所有模板，命中分数最高者即为当前状态                                   | `assets/ui` 模板图、截图正常、阈值正常                | 没有更强的页面上下文保护，谁分高认谁        |
| 主菜单进本  | 点击配置指定的关卡槽位                                                   | 固定坐标                                     | 只适合当前固定布局，槽位错了会回退到第 1 位   |
| 对话/弹窗  | 识别 `skip` 后点跳过，再尝试点 `yes`                                     | 模板命中                                     | 逻辑很窄，只覆盖这一类弹窗             |
| 助战选择   | 切职阶、找目标、翻页、刷新、找不到就回退默认位                                       | 模板、本地从者资源、默认位坐标                          | 当前已能稳定接入主链，但仍受固定布局约束      |
| 编队确认   | 找到“开始任务”按钮就点击                                                 | 模板命中                                     | 逻辑简单但完全依赖模板               |
| 加载等待   | 循环等待 `tips` 消失，再固定多等 3 秒                                      | `tips` 模板                                | 靠固定等待，不是更细判定              |
| 战斗准备   | 开启 `smart_battle` 就走智能判断，否则只在首次执行固定技能序列                       | `OCR`、战斗快照、配置                            | 判断失败时直接保守继续               |
| 选卡     | 先选可用宝具，再按普通卡 `1~5` 顺序补满 3 张                                   | `OCR`、固定卡位坐标                             | 普通卡完全不智能                  |
| 结算继续   | 两次“继续”再一次 `next`                                              | 模板命中                                     | 固定节奏，页面变化容易受影响            |
| 未知状态兜底 | 依次尝试点关闭、下一步、是/否、点屏幕等通用按钮                                      | 通用模板                                     | 能解卡死，但也有误点风险              |

### 2.3 主流程当前最值得人工核查的点

1. 状态识别是否会把相似页面误认成错误状态。
2. `UNKNOWN` 兜底点击是否会掩盖更深层识别问题。
3. 助战失败后的默认位回退是否会带来错误助战。
4. 结算页固定点击节奏是否对不同结算页样式都稳定。
5. 加载等待和助战等待是否存在过长或过短的情况。

## 3. 战斗内功能审查

### 3.1 当前已经实现的战斗内能力

| 模块 | 已实现内容 |
| --- | --- |
| `core/battle_ocr.py` | 读取前排三位 `NP`；读取敌方三个位 `HP`；读取战斗文字；读取技能角落小数字和提示文字 |
| `core/battle_snapshot.py` | 读取当前波次、敌人数、当前回合数、前排三位 `NP`、前排九个技能位是否可点 |
| `core/smart_battle.py` | 根据前排配置、从者技能资料和波次动作表，决定本回合要放哪些前排从者技能 |
| `core/battle_actions.py` | 点击从者技能、御主技能、攻击、宝具卡、普通卡、目标位 |
| `config/battle_config.yaml` | 已能配置 `OCR` 阈值、`NP` 满值、前排身份、每面动作、固定技能序列 |

### 3.2 当前判断方式

| 功能 | 当前判断方式 |
| --- | --- |
| `NP OCR` | 固定 3 个裁图区域，逐个读数字；达到 `np_ready_value` 就视为宝具可放 |
| 敌方 `HP OCR` | 固定 3 个裁图区域，按 `HP` 千分位格式整理数字 |
| 波次识别 | 固定区域读文字，再从结果里抽 `1/2/3` |
| 敌人数识别 | 先读固定数字区；读不稳时用 3 个大区域的亮度/颜色/边缘分数做回退 |
| 回合数识别 | 固定区域读文字，再抽正整数 |
| 技能可用性 | 先看技能主体综合分数；不够亮再看右下角冷却数字；还不稳再看左下角提示字；像冷却就判不能点 |
| 智能战斗决策 | 先找主打手，再按当前波次动作表逐条检查条件；满足条件且技能可点才加入动作 |
| 充能技能跳过 | 主打手 `NP` 已满时，带 `np_charge` 标签的技能直接跳过 |
| 技能目标 | 单体友方技能默认指向主打手 |
| 出卡顺序 | 先出所有已满 `NP` 的宝具，再按普通卡 `1~5` 固定顺序补满 3 张 |

### 3.3 当前缺口和半成品位

- `smart_battle` 只会规划从者技能，不会规划御主技能。
- 御主技能只能走旧的 `skill_sequence` 路线。
- `battle_actions.py` 里的 `can_use_servant_skill()` 和 `can_use_master_skill()` 只是占位。
- `battle_snapshot.py` 里的 `wave_match_threshold` 当前没有实际参与判断。
- 配置里的 `sample_mode` 目前只解析，没有真正接入逻辑。
- `coordinates.py` 里的 `SERVANT_HP_REGIONS`、`SERVANT_TRUE_NAME_REGIONS` 仍是空位。
- 没有后排、换人、替补、敌方目标智能、普通卡智能这些能力。

### 3.4 当前明显偏保守的地方

- `OCR` 读不稳时，不会积极猜，而是宁可不给结果。
- 技能主体分数不够时，会继续查角落；只要像冷却，就直接判不能点。
- 左下角提示字读不稳时，也不会放行。
- 主打手 `NP` 读不稳时，`smart_battle` 直接保守继续。
- 波次读不稳时，`smart_battle` 直接保守继续。
- 当前波次没有配置动作时，也直接保守继续。
- 主打手 `NP` 已满时，会直接倾向进入出卡，而不是继续尝试更复杂判断。
- 整套动作之间存在大量固定等待时间，明显是“稳住不乱点”优先。

### 3.5 战斗内人工审查重点

1. 技能可用性是否因为过保守导致大量漏放。
2. `enemy_count` 回退逻辑是否容易被背景或特效干扰。
3. `smart_battle` 是否过度依赖从者 `manifest` 里的技能标签。
4. 目标选择默认指向主打手，是否会在辅助技能上产生错误。
5. 固定等待时间是否明显拖慢整体流程。

## 4. 助战识别审查

### 4.1 当前主链识别方式

| 识别链 | 当前定位 | 工作方式 | 真实边界 |
| --- | --- | --- | --- |
| `embedding` | 主流程唯一方案 | 读取目标从者的正反例向量库，对三个位做遮挡排除后的整块区和脸部区双路核验，再按相似度打分 | 仍靠上下扫描，不是整页定位 |

### 4.2 `embedding` 链的真实工作方式

- 入口在 `core/support_portrait_verification.py`
- 依赖：
  - `support/generated/reference_bank.npz`
  - `support/generated/reference_meta.json`
- 每次会先把候选头像统一成 `240x260`
- 会固定忽略这些遮挡区：
  - 礼装区
  - 职阶图标
  - 攻击力
  - 星级与圣杯
  - 满破标志
- 每次识别会在固定范围内做纵向偏移扫描
- 每个位会同时看两路裁图：
  - 去遮挡后的整块人物区
  - 中心脸部区
- 反例不仅可以来自真实截图，也可以来自同职阶其他从者的 `Atlas`
- 最终要求：
  - 分数过线
  - 第一名与第二名分差过线
  - 两次确认都命中同一位

### 4.3 助战识别的当前边界

- 当前仍在做纵向扫描
- 当前没有“先在整页里找头像框”的定位步骤
- 当前建立在固定环境和固定布局上
- 当前职阶筛选只明确支持：
  - `all`
  - `berserker`

### 4.4 助战识别的前置资源

| 类别 | 位置 | 用途 |
| --- | --- | --- |
| 本地从者目录 | `local_data/servants/<className>/<slug>/` | 助战识别资源根目录 |
| 从者清单 | `manifest.yaml` | 判断资源是否存在、如何组织 |
| 原始头像图 | `atlas/faces/` | 向量库生成的原始来源 |
| 向量库 | `support/generated/reference_bank.npz` | `embedding` 链的正反例向量库，包含遮挡排除后的双路结果 |
| 向量元信息 | `support/generated/reference_meta.json` | `embedding` 链样本说明、基准尺寸和遮挡排除规则 |
| 调试截图 | `assets/screenshots/support_recognition/` | 识别失败或歧义时留档 |

如果缺少这些资源，助战相关脚本或主流程中的助战识别就起不来。

### 4.5 助战识别当前最值得人工核查的点

1. `embedding` 阈值和分差是否稳定。
2. 正反例库是否已经覆盖主要误判对象。
3. 上下扫描是否已经拖慢识别，或带来更多误判空间。
4. 找不到目标后回退默认位的风险是否可以接受。

## 5. 资源、脚本与工具审查

### 5.1 当前主要脚本入口

| 脚本 | 作用 | 当前判断 |
| --- | --- | --- |
| `scripts/build_reference_bank.py` | 生成助战向量库 | 可用，但依赖本地从者资源 |

### 5.2 助战相关脚本当前真实状态

- `build_reference_bank.py` 已支持把同职阶其他从者 `Atlas` 一起加入反例库
- `check_portrait_verifier.py` 可直接做本地图正反例核验

### 5.3 当前资源规则

当前资源规则在 `core/resources.py` 中比较明确：

- 本地从者资源只认 `local_data/servants/<className>/<slug>/`
- 原始从者图片只认 `atlas/`
- `support/` 只放运行结果和生成物，不放原始图
- 当前项目仍以 `scripts/` 作为验证和工具入口

### 5.4 人工审查重点

1. `scripts/` 里哪些脚本已经失去维护价值。
2. 是否需要把调试脚本和验证脚本职责进一步拆开。
3. 通用 `OCR` 调试脚本是否已经覆盖日常检查需求。
4. 资源生成物是否存在重复层级或命名混乱。

## 6. 运行条件与当前仓库状态

### 6.1 当前代码要求的运行前提

- Python `3.12`
- 使用 `uv` 管理依赖
- 可用的 `adb`
- `MuMu + 1920x1080`
- 本地从者资源目录：
  - `local_data/servants/<className>/<slug>/`

### 6.2 当前仓库本身的真实状态

| 项目 | 当前状态 |
| --- | --- |
| Python 依赖表 | 已收紧为当前代码真实在用的最小集合 |
| `uv.lock` | 已与 `pyproject.toml` 同步 |
| 运行环境 | 已重建并验证可用 |
| `local_data/servants` | 当前仓库里仍不存在 |
| 助战脚本可运行性 | 取决于本地从者资源是否准备齐 |

这意味着：

- 当前仓库本身已经能正常起 Python 环境
- 但在空仓库状态下，助战链和依赖本地从者资源的流程仍不能直接跑

### 6.3 当前依赖和 GPU 现状

当前依赖表主要保留：

- `adbutils`
- `numpy`
- `onnx`
- `onnxruntime`
- `opencv-python`
- `paddleocr`
- `paddlepaddle==3.0.0`
- `pillow`
- `pyyaml`

当前助战向量推理的真实状态：

- `core/portrait_embedding.py` 写死 `CPUExecutionProvider`
- 当前没有 `onnxruntime-gpu`
- 当前代码没有真正接上 GPU 推理链

所以：

- 就算机器有 GPU，当前助战 `embedding` 也不会自动走 GPU

## 8. 人工审查建议清单

建议你按下面顺序审查：

### 第一组：主链稳定性

1. 主菜单 -> 助战 -> 编队确认 -> 加载 -> 战斗 -> 结算 是否每一步都有明确失败表现。
2. `UNKNOWN` 兜底点击是否存在“把错误流程点过去”的风险。
3. 固定等待时间是否明显过长或过短。

### 第二组：战斗内判断

1. 技能可用性是否过保守。
2. `smart_battle` 是否因为条件过严而经常退回“少做动作”。
3. 目标选择默认指向主打手是否足够安全。
4. 普通卡固定顺序是否已经影响实际收益。

### 第三组：助战识别

1. `embedding` 正反例库是否足够覆盖误判对象。
2. 上下扫描是否已经成为当前识别速度和稳定性的主要问题。
3. 默认位回退是否符合你的风险预期。

### 第四组：资源与工具

1. 本地从者资源目录是否已经形成统一规范。
2. 助战生成物、模板和向量库是否存在重复层。
3. 哪些脚本是长期需要保留的，哪些只是临时校准脚本。
4. 助战观察脚本和通用 `OCR` 调试脚本的职责是否仍然清晰。

---

## 9. 审查结论汇总

一句话概括当前项目：

> 它已经是一条“固定环境下可贯通的自动刷本骨架”；助战识别主链已经补到可用，但整体仍然依赖固定环境、本地资源和持续校准。

如果继续投入，最该优先盯住的不是新功能，而是：

1. 战斗内 `OCR` 继续补稳
2. 页面状态识别误判
3. 智能战斗保守逻辑带来的漏动作
4. 脚本和资源层的清理与收口
